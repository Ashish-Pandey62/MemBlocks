# Milestone v1.1: MCP Server

**Status:** ✅ SHIPPED 2026-03-19
**Phases:** 1-4 (with inserted execution tracks 2.01 and 2.02)
**Total Plans:** 7

## Overview

Delivered a complete MCP integration layer for MemBlocks so external agents can list/create blocks, store memory (semantic/core/combined), retrieve memory (semantic/core/combined), switch active block through CLI, and read agent-facing context from MCP resources.

## Phases

### Phase 1: Foundation

**Goal**: A working MCP server that agents can connect to, list blocks, and create blocks with graceful no-active-block handling.
**Depends on**: Nothing (first phase)
**Plans**: 1 plan

Plans:
- [x] 01-01: MCP server scaffold, singleton client lifespan, active block state, and block management tools

**Details:**
- Added FastMCP stdio server and `memblocks-mcp` entry point
- Added shared active-block state file (`~/.config/memblocks/active_block.json`)
- Added `memblocks_list_blocks` and `memblocks_create_block`

## Milestone Summary

**Key accomplishments:**
- FastMCP server foundation with block management and active state
- Full store and retrieve MCP tool surface completed
- CLI and MCP resources shipped for day-to-day agent use
- Runtime gap closed for `memblocks-cli` execution

**Key Decisions:**
- Keep stdio transport for local MCP integration simplicity
- Reuse existing memory pipelines by wrapping plain text into user-message format
- Use shared JSON active-block state for cross-process CLI/MCP coordination

**Issues Resolved:**
- Broken `memblocks-cli` runtime import path during editable install
- Requirement naming drift between CLI command docs and executable command

**Issues Deferred / Known Gaps Accepted:**
- `STOR-01` and `STOR-02` remained unchecked in live requirements file at closeout; accepted as documentation/traceability debt for milestone completion

**Technical Debt Incurred:**
- Editable-install path behavior required manual correction approach; packaging flow should be hardened in a future milestone
